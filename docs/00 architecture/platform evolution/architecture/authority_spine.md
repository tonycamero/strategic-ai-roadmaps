# StrategicAI Authority Spine

The Authority Spine defines the deterministic execution architecture of StrategicAI.

It ensures that strategy generation, execution compilation, and operational roadmaps are reproducible and free from AI hallucination.

All components in the StrategicAI system must respect the rules defined here.

---

# Core Principle

StrategicAI separates strategic reasoning from execution authority.

LLMs may assist with proposal generation, but they do not define execution artifacts.

Execution artifacts must be produced deterministically by compilers.

---

# Authority Spine Pipeline

StrategicAI converts operational diagnostics into executable roadmaps through a fixed pipeline.

intake
↓
diagnostics
↓
SAS proposals
↓
SAS elections
↓
selection envelope
↓
Stage-6 execution compiler
↓
sop tickets
↓
Stage-7 graph compiler
↓
roadmap execution nodes
↓
Trust Console execution

Each step produces an artifact that becomes the authority for the next step.

---

# Canonical Artifacts

The following artifacts represent the authoritative state of the system.

sas_proposals

Generated proposals produced by Assisted Synthesis.

sas_elections

Human decisions applied to proposals.

selection_envelopes

Canonical execution intent.

sop_tickets

Deterministic operational tasks compiled from envelopes.

roadmap_execution_nodes

Runtime execution state for roadmap graphs.

These artifacts define the execution state of the system.

---

# Deterministic Compilation

Stage-6 must produce deterministic execution artifacts.

Each SOP ticket must have a stable identity.

ticketKey is derived from:

selectionEnvelopeId
capabilityId
capabilityNamespace

ticketKey = hash(
 selectionEnvelopeId +
 capabilityId +
 capabilityNamespace
)

Recompiling the same envelope must update existing tickets instead of creating duplicates.

---

# Capability Authority

Capabilities define what the system can execute.

Capabilities must be defined in the inventory registry.

Capabilities may include dependency metadata.

The compiler must validate capabilities against the registry.

Capabilities must never be inferred by LLM output.

---

# Dependency Authority

Execution dependencies must come from the inventory registry.

Dependencies must not be generated or inferred by the LLM.

The Stage-7 Graph Compiler uses registry metadata to construct execution graphs.

---

# Graph Execution Model

The Stage-7 Graph Compiler converts SOP tickets into a dependency graph.

Graph nodes represent execution tasks.

Graph edges represent capability dependencies.

Nodes are grouped into execution phases using topological sorting.

Nodes in the same phase may execute in parallel.

---

# Runtime Execution Layer

The roadmap_execution_nodes table stores the live execution state of the roadmap.

Execution nodes track:

execution status
phase order
start and completion timestamps

This layer reflects operational progress.

It does not define strategy.

---

# Provenance Tracking

Each SOP ticket must contain provenance fields.

selectionEnvelopeId
sourceFindingIds
envelopeVersion
generationEventId
projectionHash

These fields allow every execution artifact to be traced back to its originating diagnostics.

---

# System Invariants

The following invariants must always hold.

Execution artifacts must be deterministic.

Execution compilers must not use LLM inference.

Capabilities must be validated against the registry.

Dependencies must come from the registry.

Recompiling an envelope must not create duplicate execution artifacts.

The Authority Spine pipeline must remain intact.

---

# Summary

StrategicAI converts operational signals into executable strategy through a deterministic architecture.

Strategy generation may involve AI assistance.

Execution artifacts must always be generated by deterministic compilers.

This separation ensures that StrategicAI remains reliable, reproducible, and auditable.
